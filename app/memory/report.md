# 拟人 Agent 记忆模块 —— 全面分析报告

## 一、项目总览

这是一个 **AI 虚拟好友聊天系统**，模拟 QQ 单人窗口中的长期多轮对话。整个项目的核心设计理念是：**让 AI 像真人一样"记住"与用户的每一次互动，并在合适的时候自然地想起来、用得恰当**。

系统采用 **Python + FastAPI** 作为后端，**DeepSeek-V4** 作为大模型引擎，数据以 JSON 文件形式本地存储。前端是一个简单的 HTML 聊天页面。

项目的灵魂在 `app/memory/` 目录，由 **17 个模块** 各司其职，共同构成一个分层的、可演进的记忆系统。

---

## 二、系统架构：四层结构

整个应用可以理解为四个层次：

| 层次                   | 作用                             | 对应文件                                                  |
| ---------------------- | -------------------------------- | --------------------------------------------------------- |
| **入口层**             | HTTP 接口、路由、请求响应        | `main.py`                                                 |
| **服务层**             | 聊天逻辑的"总指挥"，协调所有模块 | `chat_service.py`                                         |
| **核心层（记忆模块）** | 记忆的全部生命周期管理           | `memory/` 目录下 17 个文件                                |
| **基础层**             | 存储、配置、LLM 调用、人格管理   | `storage.py`、`config.py`、`llm_gateway.py`、`persona.py` |

一次聊天请求的完整路径是：

> 用户发消息 → `main.py` 接收 → `chat_service.py` 调度 → **先召回相关记忆** → 拼装提示词发 LLM → LLM 回复 → **从回复中提取新记忆** → **质量审核** → **写入/合并/需确认** → **反思生成高级印象** → **维护（衰减/归档）** → **审计回复是否滥用了不该说的记忆** → 返回结果

---

## 三、核心记忆模块详解

### 3.1 记忆的分类体系（`schema.py`）

系统将记忆分为 **10 种长期类型** 和 **2 种人类记忆类型**：

**10 种长期记忆类型：**
- **fact（事实）**：用户说过的客观信息
- **preference（偏好）**：用户喜欢什么
- **dislike（反感）**：用户讨厌什么
- **boundary（边界）**：用户明确不想触碰的话题（雷区）
- **response_rule（回应规则）**：用户希望 AI 怎么说话
- **goal（目标/待办）**：用户有截止时间或计划的事项
- **emotion_pattern（情绪模式）**：用户在什么情境下容易产生什么情绪
- **relationship_signal（关系信号）**：用户对 AI 关系的评价（"你真懂我" vs "你太机械了"）
- **stable_impression（稳定印象）**：从多条记忆中推导出的高层次认知
- **shared_experience（共同经历）**：用户与 AI 之间的约定或共同话题

**2 种人类记忆额外类型：**
- **episodic（情景记忆）**：某次聊天中发生的具体事情（类似"我记得上周二我们聊过那个"）
- 外加上述所有长期记忆类型

每一条记忆在创建时都会自动计算以下属性：
- **importance（重要性）**：不同记忆类型有不同的基础分值，边界最高（0.94），普通事实最低（0.62）；用户确认过的、待跟进的、负面情绪的会额外加分
- **salience（显著度）**：决定记忆在召回时有多容易被"想起来"，边界和目标类最高
- **surface_policy（表露策略）**：定义了 AI 在聊天中**怎么使用**这条记忆——是默默遵守？自然跟进？还是只影响语气但不能明说？

这个分类和属性体系是整套系统运转的基础。

---

### 3.2 记忆召回：什么时候"想起来"（`recall.py`）

当用户发来一条消息时，系统需要从可能成百上千条记忆中**挑出最相关的 8 条**，这就叫"召回"。

工作方式是一个**打分排名系统**：

1. **语义匹配**：把用户消息切成词，和每条记忆的标签、内容做交集。每重叠一个词 +1.75 分
2. **用户确认过的** +0.55 分（用户亲自确认的比机器猜测的更可信）
3. **待跟进事项** +1.18 分（开放的目标/任务会得到极高优先级）
4. **情绪相关** +1.05 分（如果用户消息里包含情绪词，情绪模式和回应规则会被优先召回）
5. **旧事接续** +0.9 分（用户说了"还记得""上次""继续"等词）
6. **边界约束** +0.45 分（边界记忆始终有一定曝光度）
7. **冷却降权** -1.2 分（用了太多次又不是开环的记忆会被暂时"冷却"，避免 AI 反复提同一件事）
8. 此外还会乘以重要性、显著度、置信度的系数

只有得分 ≥ 1.05 的记忆才会被召回。这个阈值确保了**不是什么都想起来，而是想起来真正有用的**。

---

### 3.3 记忆提取：从对话中"学到"新东西（`extraction.py` + `extractors.py`）

每次聊天后，系统会从用户消息中**自动扫描**是否包含值得记住的信息。这一部分有两种实现方式：

**方式一：规则提取器（当前默认）**

纯本地运行，不需要调用 LLM，用正则表达式匹配。能识别 9 类可提取内容：

| 提取类型  | 触发条件举例                       | 置信度    |
| --------- | ---------------------------------- | --------- |
| 显式记忆  | "记住XXX"、"帮我记住XXX"           | 0.92      |
| 偏好      | "我喜欢XXX"、"我希望XXX"           | 0.78      |
| 反感      | "我讨厌XXX"、"我受不了XXX"         | 0.78      |
| 回应规则  | "以后别XXX"、"下次和我聊天时要XXX" | 0.84      |
| 边界/雷区 | 出现"雷区""不要提""别提""不想聊"   | 0.84      |
| 目标/待办 | 有时间词+任务词（"明天要交材料"）  | 0.72~0.78 |
| 情绪模式  | 检测到情绪词，推断触发情境         | 0.68      |
| 关系信号  | "你真懂我" 或 "你太机械了"         | 0.64~0.74 |
| 共同经历  | "我们约定"、"一起"、"下次继续"     | 0.74      |
| 情景记忆  | "今天/刚才" + 事件描述             | 0.58      |

**方式二：LLM 结构化提取器（可选）**

把对话发给 DeepSeek，让它以 JSON 格式返回提取的记忆。如果 LLM 调用失败，自动降级到规则提取器。当前默认使用规则提取器（配置项 `MEMORY_EXTRACTOR=rule`），这是为了保证离线可用和低成本。

两种提取器的结果最终格式完全一样，所以可以无缝切换。

---

### 3.4 记忆质量控制：不能什么都信（`quality.py`）

从对话中提取出的记忆**不是全部直接入库**的。系统会逐条审核，做出三种裁决：

- **accept（接受）**：直接写入记忆库
- **confirm（需要用户确认）**：放入待确认队列，等用户在界面上点"接受"或"拒绝"
- **reject（拒绝）**：直接丢弃

裁决的依据是一个决策树：
1. 内容太短（<4 字）→ 直接拒绝
2. 置信度 < 0.45 → 直接拒绝
3. 敏感度高 → 需要用户确认
4. 用户明确要求记住的 → 直接接受
5. 类型是低风险（偏好、回应规则、情绪模式等）且置信度 ≥ 0.62 → 直接接受
6. 类型是 fact、episodic、boundary、dislike → 需要用户确认（这些可能影响长期画像）
7. 不稳定的短期事件 + 低置信度 → 需要用户确认
8. 其他 → 默认接受

**自动接受的类型**：preference、response_rule、goal、emotion_pattern、shared_experience、relationship_signal、stable_impression

**需要用户确认的类型**：fact、episodic、boundary、dislike

这个设计体现了"对可能影响用户画像的记忆持审慎态度"的理念。

---

### 3.5 记忆生命周期管理（`lifecycle.py`）

当一条新记忆要被写入时，系统不会简单地追加，而是先检查是否**与已有记忆重复或冲突**。

**合并策略**：
- 如果新旧记忆内容相同或一方包含另一方 → 合并为一条，更新置信度、重要性、时间戳，标记为"被再次强化"
- 如果关键词重叠超过一半 → 也合并
- 用户确认状态取"或"逻辑（只要有一方是用户确认的，合并后也是）
- 证据链条最多保留最近 6 条

**冲突解决**：
- 特别针对 preference 和 dislike 之间的冲突
- 如果用户之前说"喜欢安静"，现在说"讨厌安静"，系统会把旧记忆标记为"superseded（被取代）"，而不是直接删除

**标记召回**：
- 每次记忆被召回使用后，会更新"上次使用时间"和"使用次数"，这两个指标后续会影响冷却和维护

---

### 3.6 记忆维护：防止无限膨胀（`maintenance.py`）

记忆库会不断增长，需要定期清理。当前策略：

- **情景记忆上限**：最多保留 16 条非开放的情景记忆，超出部分自动归档（标记为 archived）
- **衰减机制**：episodic 和 fact 类型的记忆如果没被用户确认、也不是待跟进事项，每次聊天轮次会降低 4% 的重要性（乘以 0.96），最低降到 0.2
- **冷却判断**：非开环、非边界、非回应规则的记忆，如果被使用了 2 次以上，在召回时会被施加 -1.2 分的冷却惩罚——这模拟了"提了太多遍的事就不要再反复提了"的社交直觉

---

### 3.7 用户纠错："你说错了，其实是……"（`correction.py`）

用户可以通过自然语言纠正 AI 的记忆。触发词包括："不是"、"记错"、"别记"、"不要记"、"忘掉"、"删掉"、"改成"、"其实是"。

支持两种纠错模式：
1. **删除**：用户说"别记那个"、"忘掉我之前说的XXX" → 匹配到的记忆标记为 deleted_by_user
2. **修正**：用户说"不是A，其实是B" → 旧记忆标记为 corrected，同时创建一条新的正确记忆（置信度 0.94，直接确认）

匹配逻辑同时使用语义降维（去掉标点和修饰词后的纯内容比对）和关键词交集。

---

### 3.8 开环关闭：自动检测"事情做完了"（`followup.py`）

当用户说"交完了"、"做完了"、"面完了"、"考完了"等完成信号时，系统会自动扫描所有"开放"状态的记忆，把与当前消息相关的待办事项标记为"已关闭"。关闭时会记录"完成描述"和"关闭时间"。

同时，`build_followup_plan` 函数会生成一个"跟进计划"，告诉 AI：
- 用户刚汇报了完成，不要继续追问 → `acknowledge_closure` 模式
- 用户没在闲聊且有待跟进记忆 → `gentle_follow_up` 模式（自然轻问）
- 用户主动说了"继续"、"上次" → `user_invited_follow_up` 模式（用户邀请接续）
- 其他情况 → 不要主动翻旧账

---

### 3.9 记忆表露策略：决定"要不要说出来"（`initiative.py`）

这是整个系统最精巧的部分之一。**记忆被召回 ≠ 可以随口说出来**。系统会为每条召回的回忆决定一个"行动"：

| 行动        | 含义                                                         |
| ----------- | ------------------------------------------------------------ |
| **obey**    | 默默遵守，绝不说出口（如边界）                               |
| **silent**  | 保持沉默，除非用户明确问起（如敏感记忆、低相关度记忆）       |
| **hint**    | 只影响语气和措辞，不能说"我记得你……"（如情绪模式、关系信号） |
| **mention** | 可以自然提起（如用户主动邀请接续、高相关度的待办）           |

决策逻辑考虑了：记忆类型、表露策略、用户是否发出了"邀请接续"的信号、记忆的召回分数、跟进计划的模式、是否在闲聊。

---

### 3.10 反思：从碎片中提炼高级认知（`reflection.py`）

这是让 AI 显得"有洞察力"的关键机制。当多条低级记忆积累到一定程度时，系统会自动生成更高层次的**稳定印象（stable_impression）**。当前有三条反思规则：

1. **情绪 + 回应规则 → 压力应对模式认知**：如果同时存在情绪模式和关于"安慰/大道理/拆解"的回应规则，生成"用户在压力时更需要先被安慰，不适合讲大道理"
2. **多个开放待办 → 陪伴式跟进认知**：如果有 2 个以上的开放目标，生成"用户习惯把未完成事项带进聊天，适合轻量陪伴式跟进"
3. **共同经历 + 关系信号 → 关系深度认知**：如果同时有共同经历和关系信号，生成"用户已把 AI 当作可以接续共同话题的长期聊天对象"

反思结果会经过去重检查（不会生成和已有印象重复的内容），然后走和普通记忆一样的质量审核流程。

---

### 3.11 用户画像：把记忆组织成"对人的理解"（`profile.py`）

`build_user_profile` 函数从所有活跃记忆中构建一个结构化的用户画像，包含：
- 各类记忆的 top 条目（按重要性、显著度、置信度排序）
- 开放事项列表（尚未完成的事）
- **关系状态评估**：根据正面/负面关系信号和共同经历数量，计算亲密度（1-5）和信任度（1-5）
- **语气指导**：综合回应规则、情绪模式、边界、稳定印象，生成一个 AI 回复时应遵循的策略列表

---

### 3.12 记忆上下文拼装：把一切发给 LLM（`context.py`）

`build_memory_context` 是记忆系统和 LLM 之间的"翻译器"。它把：
- 用户画像 → 格式化为"用户长期画像"段落
- 召回的 8 条记忆 → 每行带类型、召回原因、表露策略
- 跟进计划 → 告诉 LLM 当前该如何推进话题
- 表露计划 → 告诉 LLM 每条记忆能说到什么程度

全部拼成一个自然语言文本，注入到 LLM 的 system prompt 中。这就是"记忆在发挥作用"的最终形式。

---

### 3.13 记忆审计：事后检查 AI 有没有违规（`audit.py`）

在 LLM 生成回复之后，系统会**回头检查**回复内容是否违反了表露策略：
- **Fail 级别**：标记为 silent 或 obey 的记忆，被 AI 在回复中提起了 → 严重违规
- **Warn 级别**：标记为 hint（只影响语气）的记忆，AI 用贴标签的方式说了出来（如"你总是……"） → 过度表露
- **Warn 级别**：标记为 mention（应该自然提起）的记忆，AI 完全没有用到 → 错过了接续机会

审计结果会记录在每条 assistant 消息的元数据中，以及 generation_logs 中。

---

### 3.14 会话摘要：对话太长时的压缩（`summary.py`）

- **工作记忆**：取最近 24 条消息直接喂给 LLM（这就是"正在进行的对话"）
- **会话摘要**：当消息累积到 16 条时，自动生成一个摘要，包含话题、情绪倾向、待跟进事项、跟进建议。之后每 16 条消息追加一个新摘要。

---

### 3.15 文本工具：中文 NLP 的"瑞士军刀"（`text.py`）

这是一个纯规则的中文文本处理模块，不依赖任何 NLP 库。包含了：
- 中文分词（按双字词、英文词切分）
- 情绪识别（6 种情绪标签：压力、疲惫、低落、烦躁、积极、脆弱）
- 话题提取（从预定义关键词中匹配）
- 情感极性判断（正面/负面/脆弱/中性）
- 未完成事项检测（有时间词+动词的句子）
- 内容规范化（去修饰、去标点、去前缀，便于不同表述的同一件事被识别为同一件事）

---

### 3.16 分层视图（`views.py`）

提供了一个"仪表盘"视图，将记忆按层次统计：工作记忆几条、摘要几个、长期记忆几个、人格记忆有没有、共同经历几条、待跟进几个、关系信号几个、稳定印象几个。前端和状态 API 都会用到。

---

## 四、外围支撑层

### 4.1 LLM 网关（`llm_gateway.py`）

封装了对 DeepSeek API 的调用。支持两种模式：
- **chat**：普通聊天，temperature=0.8，带重试
- **structured**：结构化提取（JSON 输出），temperature=0.1，带重试

核心特点是**优雅降级**：如果 API 不可用（无 key、网络错误、返回异常），不会崩溃，而是返回中文自然语言格式的本地回复，比如"好，我会把这件事记进长期记忆里"。

### 4.2 JSON 存储（`storage.py`）

所有数据都存储在一个 `store.json` 文件中。特点是用读写锁保证线程安全，写入时先写临时文件再替换（防止写入一半时断电导致数据损坏）。数据结构包含：活跃会话、消息列表、摘要列表、人格版本历史、记忆列表、记忆确认队列、生成日志。

### 4.3 人格管理（`persona.py`）

用户导入背景设定文本后，系统用正则表达式从中提取：名字、关系定位、稳定性格特质、说话风格、边界规则，自动生成系统提示词。人格支持版本管理（每次导入生成新版本）。

### 4.4 聊天服务（`chat_service.py`）—— 总调度

这是整个系统的"总指挥"。一次 `chat()` 调用的完整流程是：

1. 获取当前状态快照
2. 构建记忆上下文（画像+召回+跟进+表露）
3. 拼装 LLM prompt
4. 调用 LLM 获取回复
5. 审计回复是否遵守了表露策略
6. 提取新记忆候选
7. 审核新记忆质量（接受/确认/拒绝）
8. 应用用户纠错（如果消息中有纠错信号）
9. 关闭已完成的待办
10. 标记哪些记忆被使用了
11. 写入/合并新记忆
12. 把需要确认的记忆加入队列
13. 生成反思
14. 维护记忆（衰减、归档）
15. 记录完整的生成日志
16. 返回结果

每一步都有日志记录，包括哪些记忆被用了、为什么用、哪些被创建了、哪些被修正了、哪些被删除了、哪些被衰减了等等。

## 五、关键设计亮点

1. **分层记忆架构**：工作记忆 → 会话摘要 → 长期记忆 → 人格记忆，模拟了人脑的信息加工深度递进
2. **表露策略分离**：记住 ≠ 说出来。每一条记忆都有独立的"怎么用"的策略
3. **优雅降级**：LLM 不可用时不崩溃，降级到本地规则和固定回复
4. **版本化的生成日志**：每一轮对话的所有记忆操作都有完整审计踪迹
5. **反思机制**：从碎片记忆中自动推导高层次认知，不需要人工总结
6. **自然语言纠错**：用户可以用日常表达修正 AI 的记忆，不需要操作数据库
7. **冷却机制**：防止 AI 反复提及同一件事，模拟人类的"说多了就烦了"
8. **情感感知**：情绪检测影响记忆的重要性、召回优先级、反思触发
9. **关系状态模型**：用亲密度和信任度两个数字量化 AI 与用户的关系

## 六、当前阶段和可演进方向

当前系统处于 **MVP 阶段**的成型期，已经实现的记忆能力包括：全生命周期管理（提取→审核→存储→召回→使用→审计→维护→归档），但还有一些天然可以延伸的方向：

- 记忆提取目前默认用规则（精确但覆盖面有限），切换为 LLM 提取可大幅拓宽覆盖
- 反思目前只有 3 条固定规则，可以更灵活
- 关系亲密度和信任度目前只是"计算出来展示"，还没有反向影响 AI 的回复策略
- 缺少记忆的定期"复习"机制（间隔重复）
- 可以从单用户扩展到多用户
