# 记忆模块改进思路

> 本文档针对当前记忆系统的四个核心问题，进行深入分析和方案设计。
> 不涉及具体代码修改，只阐述思路和方向。

---

## 问题一：短文本多轮对话的灵活判断

### 现状分析

当前系统对"对话结构"做了多处刚性假设：

| 假设 | 位置 | 问题 |
|------|------|------|
| 每 16 条消息生成一次摘要 | `chat_service.py:121`, `summary.py:15` | 真实对话的话题边界不按消息数量对齐 |
| 工作记忆固定取最近 24 条 | `summary.py:9` | 闲聊场景下 24 条可能只覆盖 5 分钟，重要话题下 24 条可能不够 |
| 消息 ≤ 12 字符 = 闲聊 | `followup.py:64`, `initiative.py:79` | "分手了"只有 3 字但极其重要；"今天天气真不错啊我觉得特别舒服"有 14 字但可能是纯寒暄 |
| 消息 > 18 字符才提取情景记忆 | `extraction.py:165` | 短文本中包含大量信息密度高的内容 |
| 消息 < 120 字符才提取共同经历 | `extraction.py:149` | 长消息中也可能有"我们约定" |

### 核心矛盾

系统用**消息数量**和**字符长度**这两个"表面指标"来推测**话题边界**和**信息重要性**，但这两者之间没有稳定的对应关系。网络聊天中，人们习惯用连续的短文本表达——可能 20 条消息在讨论同一个话题，也可能 1 条短消息就开启了全新话题。

### 解决思路

#### 1.1 用语义相似度替代消息计数来检测话题边界

**核心思想**：不再按"每 16 条消息"生成摘要，而是当检测到**话题转换**时才做阶段性总结。

具体做法：
- 对每条用户消息计算嵌入向量（embedding）
- 维护一个"当前话题"的累积向量（可以是最近 N 条消息向量的加权平均）
- 当新消息与当前话题的余弦相似度低于某个阈值（如 0.6）时，判定为话题转换
- 话题转换时触发：生成上一话题的摘要、清理短期上下文、重新评估需要召回的记忆

这样带来的好处：
- 一个持续 50 条消息的话题不会被强行切成 3 段摘要
- 5 条消息就结束的话题不会被等到 16 条才总结
- 摘要的语义完整性大幅提升

#### 1.2 用"信息密度"替代字符长度来判断消息重要性

**核心思想**：不因为消息短就判定为闲聊，而是看消息的**语义负载**。

具体做法：
- 对每条用户消息用轻量方式评估信息密度：
  - 是否包含新实体（人名、时间、地点、事件）？
  - 是否包含情感词？
  - 是否包含决定/承诺/请求？
  - 与前一条消息的语义差异有多大？
- 高密度短消息（"我分手了"、"明天面试"、"别记这个"）正常触发记忆提取和跟进
- 低密度长消息（"哈哈哈哈笑死我了今天真的太搞笑了你知道吗"）可以降低处理优先级

判断"是否为闲聊"不再看 `len(text) <= 12`，而是看**语义增量**——这条消息让对话往前推进了多少。

#### 1.3 用时间窗口对短消息进行"聚团"

**核心思想**：30 秒内连续发送的 5 条短消息本质上是一个"话轮"，应当合并处理。

具体做法：
- 维护一个消息缓冲窗口（如 60 秒或用户停止输入 3 秒后）
- 将时间上紧密相邻的消息合并为一个"逻辑话轮"
- 合并后的话轮作为记忆提取和意图判断的基本单位
- 这样既保留了短文本交流的自然节奏，又避免了"每条都判断一次"的碎片化

#### 1.4 动态工作记忆窗口

**核心思想**：工作记忆不总是 24 条，而是根据话题的深度和复杂度动态伸缩。

具体做法：
- 简单寒暄：6-8 条足够（节省 token）
- 日常聊天：默认 24 条
- 深度话题（检测到高频情感词/复杂逻辑）：扩展到 36-48 条
- 判断依据：话题切换频率、情感强度、用户主动说"继续""上次"等信号

## 问题二：JSON 文件作为记忆存储介质的长期适用性

### 现状分析

当前使用单一 `store.json` 文件存储所有数据：消息、记忆、人格、摘要、确认队列、生成日志。每次读写操作都是**全量序列化/反序列化**，通过文件锁保证线程安全。

这在 MVP 阶段（几百条记忆、几千条消息）是完全合理的。但当记忆库增长到一定规模时，会出现以下问题：

| 问题 | 影响 |
|------|------|
| **无索引，全量扫描** | 召回记忆时需要遍历所有活跃记忆，O(n) 复杂度 |
| **全量读写** | 每轮聊天都要读整个 JSON、改几个字段、写整个 JSON |
| **文件膨胀** | 已删除、已归档、已取代的记忆物理上仍占用空间 |
| **无查询能力** | "所有 open=True 且 type=goal 的记忆"必须 Python 遍历 |
| **并发受限** | 基本的线程锁，无法支撑多用户或多会话 |
| **无事务** | 写入中间崩溃可能丢数据（当前用 tmp 文件缓解） |
| **无全文搜索** | 记忆召回靠 Python 层面的关键词交集，非常低效 |

### 核心判断

**JSON 文件对 MVP 是正确的选择**——它让开发、调试、数据检查变得极其简单。但面向长期使用，需要一个**渐进式升级路径**，而不是一次性推倒重来。

### 解决思路

#### 2.1 推荐路径：JSON → SQLite + FTS5 → SQLite + 向量搜索

**第一阶段（立即可做）：迁移到 SQLite**

SQLite 是嵌入式数据库，不需要独立服务，和 JSON 文件一样零配置，但提供了：
- **B-Tree 索引**：在 type、status、open、created_at、importance 上建索引，召回从全量扫描变为索引查询
- **FTS5 全文搜索**：SQLite 内置的中文分词 + 全文索引，替代当前的 `tokens()` 关键词交集。查询"焦虑压力"能匹配到"最近压力很大很焦虑"，且支持 BM25 相关性排序
- **真正的增量更新**：只写变化的行，不读写整个数据库
- **事务保证**：SQLite 的 WAL 模式提供崩溃恢复
- **数据不丢失**：旧数据可完整迁移，JSON → SQLite 是一次性的导入

**第二阶段（后续）：引入向量搜索**

在 SQLite 基础上增加嵌入向量存储：
- 使用 `sqlite-vec` 扩展（SQLite 官方向量扩展，零配置）或 LanceDB（嵌入式向量数据库）
- 对每条记忆计算 embedding，存储到向量表
- 召回时同时查询：FTS5（关键词精确匹配）+ 向量相似度（语义匹配）
- 混合排序：`最终得分 = 0.4 × BM25得分 + 0.6 × 向量相似度`
- 这解决了一个关键问题：用户说"我最近睡不好"，但记忆库里存的是"最近失眠严重"——关键词不重叠但语义一致，向量搜索能召回，FTS5 不能

**为什么不一步到位上向量数据库（ChromaDB/Milvus/Qdrant）？**
- 对于单人/少人使用的聊天系统，这些是过度设计
- 它们需要独立进程、额外运维
- SQLite + sqlite-vec 已经是单文件、零运维的向量方案
- 保持和 MVP 阶段一样的"一个文件，拖走即用"体验

#### 2.2 表结构设计思路

不是简单地把 JSON 的 key 映射为列，而是按照**查询模式**来设计表：

```text
memories 表
├── id, type, content (TEXT, FTS5 索引)
├── importance, salience, confidence (REAL, 有索引)
├── status (TEXT: active/archived/deleted/superseded, 有索引)
├── open (INTEGER, 有索引)
├── created_at, updated_at, last_used_at (TEXT/ISO)
├── tags (TEXT, 逗号分隔或关联表)
├── evidence (关联表或 JSON 列)
└── embedding (BLOB, 第二阶段)

messages 表
├── id, session_id, role, content, created_at
└── meta (JSON)

generation_logs 表
├── id, created_at, purpose, provider, model
├── degraded, elapsed_ms, error
└── prompt_manifest (JSON)
```

关键设计原则：
- **高频查询字段建索引**：status、type、open、importance
- **低频字段用 JSON 列**：meta、evidence、tags——保持灵活性，不需要 JOIN
- **FTS5 虚拟表**：只对 content 列建全文索引
- **embedding 独立表**：避免加载记忆列表时拉取大 blob

#### 2.3 迁移策略

- 写一个迁移脚本 `migrate_json_to_sqlite.py`，读取现有 `store.json`，逐表导入
- 迁移后保留 `store.json` 作为备份，不删除
- 存储层抽象为接口（`StorageBackend`），`JsonStore` 和 `SqliteStore` 实现同一接口
- 通过配置切换：`STORAGE_BACKEND=json` 或 `sqlite`
- 这样可以在两个后端之间做 A/B 对比，确认 SQLite 正确后再移除 JSON

## 问题三：量化参数设计的合理性及调优方法

### 现状分析

经过全面审计，当前系统中有 **80+ 个硬编码的数值参数**，分布如下：

| 类别 | 数量 | 典型例子 |
|------|------|----------|
| 召回评分权重 | 12 个 | 关键词重叠 ×1.75、待跟进 +1.18、冷却惩罚 -1.2 |
| 重要性基值 | 11 个 | boundary=0.94、fact=0.62、默认 0.5 |
| 显著度调整 | 5 个 | open_item +0.16、boundary/goal +0.22 |
| 提取置信度 | 11 个 | 显式记忆 0.92、情景记忆 0.58 |
| 质量阈值 | 4 个 | 拒绝线 0.45、自动接受线 0.62 |
| 维护参数 | 4 个 | 衰减率 0.96、衰减下限 0.2、情景上限 16 |
| 上限/截断 | ~20 个 | 召回 8 条、画像每类 6 条、语气指导 6 条 |
| 合并参数 | 5 个 | 置信度上限 0.98、合并加分 +0.05 |
| 表露阈值 | 2 个 | mention ≥ 4.5、hint ≥ 4.2 |

这些数值**都是拍脑袋定的**，没有经过系统调优。当前完全无法回答以下问题：
- "如果把待跟进事项的召回加分从 1.18 降到 0.9，用户会觉得 AI 更不贴心了还是更不烦人了？"
- "自动接受的置信度阈值 0.62 是太松了还是太严了？有多少该确认的记忆被悄悄接受了？"
- "衰减率 0.96（每轮降 4%）意味着一条 fact 记忆从 0.62 衰减到 0.45（被拒绝线）需要多少轮？"

### 核心矛盾

参数分散在 17 个文件中，没有中心化的参数管理，没有反馈回路，无法知道一个参数的改变产生了什么影响。

### 解决思路

#### 3.1 参数集中化管理

**第一步**：将所有可调参数提取到一个配置文件 `memory_params.yaml`（或一个 `MemoryParams` 数据类）中。

```yaml
recall:
  min_score_threshold: 1.05
  token_overlap_multiplier: 1.75
  confirmed_bonus: 0.55
  open_item_bonus: 1.18
  emotion_match_bonus: 1.05
  history_continuation_bonus: 0.90
  boundary_bonus: 0.45
  cooldown_penalty: 1.20
  importance_weight: 0.78
  salience_weight: 0.46
  confidence_weight: 0.35
  max_recalled: 8

importance:
  base_values:
    boundary: 0.94
    response_rule: 0.88
    goal: 0.84
    # ... 全部类型

quality:
  reject_min_confidence: 0.45
  auto_accept_min_confidence: 0.62
  low_stability_confirm_threshold: 0.70

maintenance:
  decay_multiplier: 0.96
  decay_floor: 0.20
  max_ephemeral: 16
  cooldown_use_threshold: 2

# ... 所有其他参数
```

这样做的好处：
- 所有参数一目了然，不再散落各处
- 改参数不需要改代码，改完重启即可生效
- 可以为参数写文档注释（这个值控制什么、调高/调低的影响）
- 可以保存多套参数方案做对比

#### 3.2 建立隐式反馈回路

当前系统已经在记录 `use_count`（记忆被使用的次数）和 `last_used_at`，但这些数据没有被用于参数调优。可以扩展这个思路：

**可采集的隐式反馈信号**：
| 信号 | 含义 | 可用于调优 |
|------|------|-----------|
| 用户在被追问后回复了 | 跟进策略有效 | 调高 open_item_bonus |
| 用户在被追问后转移话题 | 跟进太频繁/不合适 | 调低 open_item_bonus，或调高冷却惩罚 |
| 用户说"别记""记错了" | 提取或合并出错 | 调整该类型的提取置信度 |
| 用户点了"接受确认" | 质量审核太保守了 | 该类型可考虑从 CONFIRM 移到 AUTO_ACCEPT |
| 用户点了"拒绝确认" | 质量审核合适 | 不需要调整 |
| 某条记忆被多次召回但从没被用过 | 召回评分虚高 | 该类型的召回权重需要降低 |
| 用户主动说"还记得""上次" | 冷却惩罚可能太激进 | 调高 cooldown_use_threshold |
| AI 回复后用户继续聊同一话题 | 表露策略有效 | 不需要调整 |
| AI 表露记忆后用户沉默或换话题 | 表露不当 | 调高 hint/mention 的 recall_score 阈值 |

**实现方式**：
- 在 `generation_logs` 中增加 `feedback_signals` 字段
- 定期运行一个离线的"参数效果分析脚本"
- 脚本读取日志，统计各项信号的变化趋势，输出参数调整建议

#### 3.3 参数配置档（Profile）

不同的使用场景需要不同的参数组合。与其让用户逐个调参，不如预设几套"性格档位"：

| 档位 | 特点 | 参数调整方向 |
|------|------|-------------|
| **谨慎型** | 少犯错，多确认，少主动 | 提高质量阈值、降低召回上限、缩短跟进 |
| **平衡型（默认）** | 当前参数 | 不变 |
| **主动型** | 更贴心，更主动，允许偶尔犯错 | 降低确认阈值、提高召回数、增加跟进频率 |
| **怀旧型** | 多提旧事，多接续 | 降低冷却惩罚、提高情景记忆重要性、放宽旧事接续触发 |

用户可以在设置中选择档位，系统加载对应的参数覆盖文件。

#### 3.4 参数校准数据集

思路：积累一批"标准答案"——人工标注的记忆决策案例，用它们来系统性地校准参数。

具体做法：
- 收集 50-100 个"用户消息 → 应该提取什么记忆 → 应该召回哪些旧记忆 → AI 应该怎样使用它们"的四元组
- 每个四元组由人标注"理想结果"
- 对于每一组参数，用脚本模拟运行，计算与理想结果的偏差
- 用网格搜索或贝叶斯优化找到使偏差最小化的参数组合

这个方案比较重，适合在系统稳定运行一段时间、积累了一定用户反馈后执行。

#### 3.5 参数可解释性

每个参数应该有文档说明，格式如：

```yaml
recall.open_item_bonus:
  value: 1.18
  description: "待跟进事项的召回加分。这是所有单项加分中最高的，因为我们希望 AI 优先记住用户还没完成的事。"
  sensitivity: high
  range: [0.5, 2.0]
  effect_of_increasing: "AI 会更频繁地提及未完成事项，更贴心但也更可能显得催促"
  effect_of_decreasing: "AI 会更少主动跟进，用户需要自己提起旧事"
```

## 问题四：关键词命中机制的灵活性不足

### 现状分析

经过全面审计，当前系统中有 **约 40 组硬编码的中文关键词列表**，直接影响记忆的提取、分类、召回、纠错、跟进、审计等所有环节。

关键问题不在关键词的数量（40 组、200+ 个词覆盖了常见场景），而在于：

1. **同义不同词**：用户说"搞定了"，系统不认识（COMPLETION_WORDS 里只有"做完""完成""结束"）
2. **新兴表达**："emo了" = 低落、"破防了" = 脆弱、"摆了" = 放弃/拖延，这些词不在词典里
3. **否定/反讽**："我真是太开心了（反话）"被当作积极情绪处理
4. **跨语言混合**："我今天真的 very tired"——英文情绪词完全不被识别
5. **组合语义**：单独一个"累"字可能是"累了吗"（关心）或"好累"（疲惫），语境不同含义不同

### 核心矛盾

关键词匹配的精度（precision）尚可，但召回率（recall）取决于词典的完备性，而词典永远赶不上语言的变化。此外，关键词完全无法处理**语义等价**和**上下文依赖**。

### 解决思路

这里的关键设计原则是：**分层处理，逐步升级**。不是把所有关键词一刀切替换为 LLM，而是：

- **高频 + 明确 + 低延迟要求的场景**：保留关键词，但用嵌入向量扩展同义词
- **低频 + 模糊 + 高价值场景**：使用 LLM 做语义判断
- **中间地带**：关键词初筛 → LLM 确认

#### 4.1 分类处理：哪些适合升级、哪些保留关键词

**适合保留关键词的场景（规则够用，升级不划算）**：
- 话题提取（13 个词）：工作/学习/游戏/朋友……这些是领域标签，关键词足够
- 显式记忆命令（"记住""帮我记住"）：用户明确指令，不需要语义理解
- 正则提取（"我喜欢XXX""我讨厌XXX"）：正则已经足够精确

**适合升级为语义匹配的场景**：
- 完成信号检测（"做完了"←"搞定了"←"收工了"←"终于交了"）
- 纠错意图检测（"别记这个"←"不要记"←"删了这条"←"这个不对"）
- 情绪识别（尤其是复合情绪和反讽）
- 关系信号检测（用户是在亲近还是在疏远）

**适合升级为 LLM 分类的场景**：
- 跟进模式判断：用户这句话是真的在闲聊，还是在暗示想让 AI 问下去？
- 边界/雷区判断：用户说"最近不太想聊工作"，这是一个边界还是一次性的情绪表达？
- 共同经历判断：用户说"和上次一样"，是否需要召回上次的共同经历？

#### 4.2 嵌入向量同义词扩展（低成本、高收益）

这是解决关键词覆盖不足的**最轻量方案**：

**做法**：
1. 取每个关键词列表（如 COMPLETION_WORDS = ["做完了", "完成了", "结束了"...]）
2. 用嵌入模型计算这些词在语义空间中的位置，得到一个"已完成"语义簇的中心向量
3. 对中文常用动词计算嵌入，找出与中心向量余弦相似度 > 0.8 的所有词
4. 自动扩展关键词列表

**效果预估**：
- "做完了" → 自动发现："搞定了"、"收工了"、"完事了"、"结了"、"办妥了"、"OK了"
- "焦虑" → 自动发现："心慌"、"不安"、"紧张兮兮"、"慌"、"压力山大"
- "别记" → 自动发现："不用记"、"这个不要"、"别存"、"忽略这条"

**优点**：不改变系统架构，不增加推理延迟，只是让已有的关键词列表更"胖"。

**局限**：仍然无法处理否定、反讽、上下文依赖。

#### 4.3 LLM 意图分类器（中等成本、大幅提升精度）

对于**高价值但关键词难以覆盖**的判断，使用 LLM 做一个轻量的分类调用：

**设计方案**：
- 把当前需要判断的意图打包成一个结构化的分类 prompt
- 一次 LLM 调用同时输出多个分类结果
- 使用低 temperature（0.1），要求严格 JSON 输出
- 结果被缓存，同一轮对话不重复调用

**一次调用可同时判断**：
```json
{
  "has_completion_signal": true,
  "completion_target": "面试",
  "has_correction_intent": false,
  "primary_emotion": "焦虑",
  "secondary_emotion": "疲惫",
  "valence": "negative",
  "is_casual_chat": false,
  "has_followup_invitation": false,
  "topics": ["面试", "工作"],
  "unfinished_items": ["准备面试材料"]
}
```

**调用时机**：
- 不是在每条用户消息上都调用（太贵太慢）
- 而是在**关键词初筛命中**后才调用做确认
- 或者**关键词未命中但消息包含高情感密度词**时作为补充

**降级策略**：
- LLM 不可用时，退回到关键词匹配（当前的逻辑）
- 保证系统在任何情况下都能工作

#### 4.4 嵌入向量 + 轻量分类器（中间方案）

如果不想依赖 LLM 调用的延迟和成本，可以使用本地嵌入模型 + 逻辑回归/SVM 做分类：

**做法**：
1. 收集一批标注数据（用户消息 → 情绪/意图/话题标签）
2. 用嵌入模型将消息转为向量
3. 训练一个轻量分类器（scikit-learn LogisticRegression，推理极快）
4. 部署为本地推理，无网络调用，毫秒级延迟

**优点**：快速、离线、低成本
**缺点**：需要标注数据，准确率不如 LLM，无法处理复杂推理

#### 4.5 推荐的分阶段升级路径

| 阶段 | 内容 | 效果 |
|------|------|------|
| **Phase 1（立即可做）** | 嵌入向量扩展所有关键词列表 | 同义词覆盖提升 50-80%，零延迟增加 |
| **Phase 2（近期）** | LLM 意图分类器替代关键判断点 | 完成信号、纠错意图、情绪识别准确率大幅提升 |
| **Phase 3（中期）** | 训练本地轻量分类器 | 将 Phase 2 中已验证的判断逻辑沉淀为快速本地模型 |
| **Phase 4（远期）** | 端到端记忆提取全部由 LLM 完成 | 当 DeepSeek 的结构化提取足够稳定和便宜时 |

## 总结：四个问题的优先级和依赖关系

```
优先级排序（从高到低）：

1. 参数集中化管理（问题三的 3.1）
   理由：改动最小、收益立竿见影、为后续调优打基础
   依赖：无
   工作量：小（1-2 个文件，纯配置提取）

2. 嵌入向量同义词扩展（问题四的 4.2）
   理由：低成本解决最大的关键词覆盖痛点
   依赖：需要一个中文嵌入模型（可用 DeepSeek 的 embedding API 或本地模型）
   工作量：中（需要离线计算一次同义词表 + 修改 text.py）

3. 语义话题边界检测（问题一的 1.1）
   理由：解决"每 16 条消息强制摘要"的核心体验问题
   依赖：需要 embedding（可与问题四共用）
   工作量：中（修改 summary.py + context.py）

4. 信息密度评估替代字符长度（问题一的 1.2 + 1.3）
   理由：解决短消息误判问题
   依赖：可与问题四的意图分类器合并实现
   工作量：中

5. JSON → SQLite 迁移（问题二）
   理由：性能提升最明显，但改动最大，需要充分测试
   依赖：上述参数集中化（因为迁移时会触及所有模块）
   工作量：大（涉及 storage.py 重写 + 所有调用方的适配）

6. 隐式反馈回路（问题三的 3.2）
   理由：需要积累数据后才能发挥作用
   依赖：SQLite（查询日志方便很多）
   工作量：大

7. LLM 意图分类器（问题四的 4.3）
   理由：大幅提升精度，但增加延迟和成本
   依赖：网关已支持 structured 调用
   工作量：中
```
