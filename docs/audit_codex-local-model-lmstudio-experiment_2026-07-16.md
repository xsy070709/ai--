## 三、三轮有效迭代

### 第一轮：结构化记忆安全边界 — 已完成

**发现**：模型在规则未识别纠错时仍可单独输出 `delete/correct` 并触发记忆变更；同时 Python 会把字符串 `"false"` 当成真值，使模型输出有机会把候选误标为已确认或待跟进。

**改进**：

- 完成、纠错和删除等高影响动作必须由规则分类器提供证据；模型只能补充低风险字段。
- `is_user_confirmed` 改为从用户原文中的“记住、以后、雷区”等明确表达推导，不采信模型自报的确认状态。
- 布尔值只接受真正的 JSON boolean；列表只接受 JSON array；异常成员安全忽略。
- 非目标类型、字符串 open 标记和异常选择值继续保持拒绝或安全回落。

**结果**：新增模型越权纠错、字符串布尔值、标量列表和伪确认回归；全量测试 `194 passed`。

### 第二轮：本地调用失败预算 — 已完成

**发现**：LM Studio 原先复用远端 `LLM_TIMEOUT_SECONDS=30` 与 `LLM_MAX_RETRIES=2`，启用同步结构化工作流后，一次本地故障理论上可阻塞约 90 秒，与试点“不影响前台响应”的目标冲突。

**改进**：

- 增加 `LOCAL_STRUCTURED_TIMEOUT_SECONDS=12` 和 `LOCAL_STRUCTURED_MAX_RETRIES=0`。
- 本地 provider 使用独立预算，默认一次失败最多等待约 12 秒后进入规则回退。
- memory extract 和 memory intent 的严格 JSON Schema 增加 `additionalProperties: false`，减少额外字段漂移。
- README 与 `.env.example` 同步说明真实调用边界。

**结果**：新增独立超时、独立重试次数和严格 Schema 回归；全量测试 `195 passed`，记忆校准 `score 1.0`。

### 第三轮：运行时路由与降级可观测性 — 已完成

**发现**：旧 `/api/llm/health` 只报告主聊天 DeepSeek 配置，无法说明记忆抽取和意图分类实际使用 rule、DeepSeek 还是 LM Studio，也无法看到最近是否降级；无密钥本地回退没有进入请求状态记录。

**改进**：

- 健康接口分别报告 chat、memory extract、memory intent 的实际 mode、provider、model 和 configured 状态。
- LM Studio 未调用时 `available=null`，调用后根据最近结果更新为 `true/false`。
- 无 DeepSeek key 的 chat/structured 回退也记录为降级请求。
- 健康摘要只暴露时间、provider、模型、耗时和错误，不携带提示词、用户原文或模型回复正文。
- 保留旧的扁平健康字段，避免已有调用方破坏。

**结果**：新增有效路由、未知状态、最近降级和健康摘要脱敏回归；全量测试 `197 passed`，记忆校准 `score 1.0`。

## 五、剩余风险与非阻塞项

1. 显式启用 `MEMORY_EXTRACTOR=lmstudio` 或 `MEMORY_INTENT_CLASSIFIER=lmstudio` 时，本地结构化调用仍位于当前聊天请求链路；本轮限制了最坏等待时间，但真正“不影响前台”仍需要后台 shadow queue。
2. 本轮没有重新连接真实 LM Studio 模型执行在线 smoke test；自动化验证使用可控的 provider stub，原始试点结果仍保存在 `docs/lmstudio_backend_pilot_results_2026-07-04.json`。
3. Nomic embedding 在原始中文负例上相似度偏高，仍不应替换默认 `local-hash-v1` 召回信号。
4. 健康接口采用被动观测：LM Studio 尚未调用时返回 `available=null`，不会主动探测服务或触发模型加载。
5. 数值型环境变量仍依赖合法配置；生产化时可进一步加入集中式配置校验与启动错误提示。

以上风险不阻塞本次试点合并，但第 1、2 项应在把本地模型默认接入用户请求前完成。
