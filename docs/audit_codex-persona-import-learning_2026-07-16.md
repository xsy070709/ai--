## 三、审计发现与完成情况

### C1：事件 CRUD 可跨人格实体写入 — 已修复

**之前**：事件列表只返回当前人格的数据，但更新、删除、解决和确认按全局 `event_id` 查找。持有另一个人格的事件 ID 时，可以跨实体修改；“解决事件”还会把目标事件生成的记忆错误标记为当前人格。

**之后**：所有事件写操作同时校验当前 `persona_entity_id`。跨实体调用统一返回未找到，不改变事件或记忆。

### C2：记忆删除与确认可跨人格实体写入 — 已修复

**之前**：记忆列表和待确认列表按当前人格过滤，但按 ID 删除、接受或拒绝时没有再次校验实体边界。

**之后**：删除和确认写操作在同一存储事务内校验当前人格，避免“读隔离、写不隔离”。

### H1：删除好友遗留事件 — 已修复

删除人格实体现在会同时清理其 sessions、persona versions、memories、confirmations、generation logs 和 persona events。测试明确断言被删实体不再残留事件。

### H2：事件与关联记忆可能重复或失步 — 已修复

事件记忆新增稳定的 `source_event_id`。事件创建、编辑、确认和解决采用同一条关联记忆的更新路径；解决事件不再追加重复记忆，删除事件会删除对应关联记忆。

### H3：导入会话目标实体消失时假成功 — 已修复

开始导入时拒绝不存在的人格实体；导入过程中实体被删除时，确认操作返回明确的实体不存在错误，不再删除导入会话并返回缺少 persona 的成功响应。

### H4：模型等待期间实体被删除会污染替代人格 — 已修复

聊天开始时固定 session 和 persona entity。模型返回前原 session 被删除或改绑时，服务拒绝提交并由 API 返回 `409`；回复、记忆和生成日志不会落入自动创建的替代人格。

## 五、剩余风险与非阻塞项

1. 导入向导会话仍仅保存在进程内，服务重启后未确认内容会丢失。
2. 本地规则学习适合作为故障兜底，无法替代模型对复杂多人叙事和隐含关系的理解。
3. 本轮完成了自动化服务层验证和前端语法检查，没有连接真实 DeepSeek 模型执行端到端浏览器验收。
4. 当前系统定位仍是单用户 MVP；若扩展为多账户服务，还需要账户级授权边界，不能只依赖人格实体 ID。
5. 旧数据中已经存在的、尚未带 `source_event_id` 的事件记忆不会被自动猜测关联，以避免错误删除历史记忆；新建事件已使用稳定关联字段。

以上风险不阻塞本次 MVP 合并，但第 1、3、4 项应在生产化前处理。
